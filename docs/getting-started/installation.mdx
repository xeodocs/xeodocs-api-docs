Welcome to the installation guide for XeoDocs, the innovative platform on xeodocs.com that uses AI to translate open-source technology documentation, typically from English into other languages. This guide covers setting up the backend microservices (built in Golang), database (PostgreSQL), message queue (RabbitMQ), and local file system storage using Docker for a seamless development or production environment with asynchronous task processing.

## Prerequisites

Before installing XeoDocs, ensure you have the following:

- **Docker and Docker Compose**: Version 20.10+ for containerization.
- **Golang**: Version 1.21+ (for building from source, optional if using pre-built images).
- **Git**: For cloning repositories.
- **GitHub Account**: An organization or personal GitHub account for forking source repositories and publishing translated content. Configure access via a GitHub App for security.
- **API Keys**: For AI translations (e.g., xAI or OpenAI API key) â€“ set as environment variables.
- **Hardware**: At least 4GB RAM and 2 CPU cores for local dev; more for production.

Clone the backend repository:
```bash
git clone https://github.com/xeodocs/xeodocs-backend.git
cd xeodocs-backend
```

## GitHub Integration

XeoDocs requires GitHub API access to fork source repositories, create language-specific branches, and publish translated static content to new repositories. For maximum security, use a GitHub App instead of a personal access token (PAT), as it provides fine-grained permissions and scoped access without sharing personal credentials.

### Setting Up a GitHub App

1. **Create a GitHub App**:
   - Go to your GitHub account settings (or organization settings if using an org).
   - Navigate to "Developer settings" > "GitHub Apps" > "New GitHub App".
   - Fill in:
     - **App name**: XeoDocs API
     - **Homepage URL**: https://xeodocs.com (or your domain)
     - **Webhook URL**: Leave blank (not needed for API-only access)
     - **Permissions**:
       - Repository permissions:
         - Contents: Read and Write
         - Metadata: Read
         - Pull requests: Read and Write (for forking)
       - Organization permissions (if applicable):
         - Members: Read (to manage repos in org)
     - **Where can this GitHub App be installed?**: Only on this account (or organization)
   - Generate a private key and download it (e.g., `xeodocs-app.pem`).

2. **Install the App**:
   - On the app page, click "Install App" and select the account/organization where repos will be forked and created.

3. **Configure Environment Variables**:
   - Add the following to your `.env` file:
     ```
     GITHUB_APP_ID=your_github_app_id  # Numeric ID from app settings
     GITHUB_PRIVATE_KEY=-----BEGIN RSA PRIVATE KEY-----\n...your_private_key...\n-----END RSA PRIVATE KEY-----  # Base64-encoded or multiline
     GITHUB_INSTALLATION_ID=your_installation_id  # From installation URL or API
     ```
   - The private key should be the full PEM content, possibly base64-encoded for single-line storage.

This setup ensures secure, token-less authentication using JWTs generated from the private key, scoped only to necessary permissions.

## Local Development Setup

Use `docker-compose.dev.yml` for a local environment with volume mounts for hot-reloading.

1. **Build and Start Services**:
   ```bash
   docker compose -f docker-compose.dev.yml up --build -d
   ```
   This starts:
   - API Gateway (port 12020)
   - Auth Service (internal)
   - Project Service (internal)
   - Repository Service (internal)
   - Translation Service (internal)
   - Build Service (internal)
   - Logging Service (internal)
   - Scheduler Service (internal)
   - Worker Service (internal)
   - Analytics Processor Service (internal)
   - Analytics Service (internal)
   - PostgreSQL (internal)
   - RabbitMQ (internal)
   - Kafka (internal)
   - InfluxDB (internal)

2. **Database Migrations**:  
   Automatically run by microservices on first startup using [Goose](https://github.com/pressly/goose).

3. **Access the API**:
   - Gateway: `http://localhost:12020/v1`
   - Repositories are stored locally with mounted volumes (e.g., `./repos:/repos` for processing).
   - Test login: POST to `/v1/auth/login` with default credentials (seed an admin user if needed).

4. **Frontend Integration**:
   For the Next.js admin panel, refer to the separate frontend repo setup. Point it to `http://localhost:12020/v1`.

## Production Setup

The Docker images are built in GitHub Actions and pushed to Docker Hub.

**Option 1: Docker Compose**

Use `docker-compose.prod.yml` for example in a remote VPS.

```bash
docker compose -f docker-compose.prod.yml up -d
```

**Option 2: Kubernetes**

Use Helm charts or YAML manifests to deploy services as Deployments.


## Troubleshooting

- **DB Connection Issues**: Verify `.env` credentials and container networks.
- **AI API Errors**: Check the API key and network access.
- **GitHub API Errors**: Ensure the GitHub App is properly installed, permissions are set, and environment variables (GITHUB_APP_ID, GITHUB_PRIVATE_KEY, GITHUB_INSTALLATION_ID) are correct. Check app logs for JWT or token issues.
- **Git Clone Failures**: Ensure public repos and handle rate limits.
- **Kafka Connection Issues**: Verify KAFKA_BROKERS and ensure the Kafka container is running and accessible.
- **InfluxDB Errors**: Check INFLUXDB_URL, INFLUXDB_TOKEN, INFLUXDB_ORG, and INFLUXDB_BUCKET for correctness and ensure the InfluxDB container is initialized.
- **Logs**: Use the Logging Service or container logs (`docker logs <container>`).
